# APM

Alternatives Names: Application Performance Monitoring; Distributed Tracing; Observability
Resource Link: https://opentelemetry.io/docs
Category: Observability Concept
Programming Language: Python / Go / Java / TypeScript (language-specific agents)
Description: The practice and tooling category for tracking service-level metrics: request latency, error rates, throughput, and resource usage across distributed services. Typically includes distributed tracing to follow a request across microservices. Implemented via agents or OpenTelemetry instrumentation.
Role in VC Assistant Infrastructure: General observability layer for FastAPI, Celery workers, and document pipeline services. Tracks request latency per endpoint, DB query performance, queue depth, and error rates. Distinct from LLM observability (LangSmith/Langfuse) which is prompt/token-aware — both layers are needed.
Potential Alternatives (with main comparison): Datadog APM: most complete but expensive. New Relic: similar to Datadog, competitive pricing. Dynatrace: AI-powered anomaly detection, enterprise-focused. Jaeger (open-source): distributed tracing only, no metrics. Zipkin: lightweight open-source tracing. OpenTelemetry: vendor-neutral instrumentation standard that feeds into any of the above backends — recommended as the instrumentation layer regardless of backend choice.
Deployment Notes & File Structure: APM is a concept — agent deployed alongside app. Recommended structure:
app/
  observability/
    tracing.py            # OTel tracer setup
    metrics.py            # custom metric definitions
    middleware.py         # FastAPI tracing middleware
infra/
  otel-collector/
    otel-config.yaml      # collector pipeline config
    deployment.yaml       # K8s collector deployment
Short File/Config Template: # tracing.py (OpenTelemetry)
from opentelemetry import trace
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
from opentelemetry.instrumentation.sqlalchemy import SQLAlchemyInstrumentor
FastAPIInstrumentor.instrument_app(app)
SQLAlchemyInstrumentor().instrument(engine=engine)
Stack & Framework Integration: Instrumented at the FastAPI application level via auto-instrumentation (ddtrace or opentelemetry-instrumentation-fastapi). SQLAlchemy, Redis, and Celery also auto-instrumented. Traces flow through API → worker → DB with a single trace ID. OpenTelemetry exporters allow switching backends without code changes.
