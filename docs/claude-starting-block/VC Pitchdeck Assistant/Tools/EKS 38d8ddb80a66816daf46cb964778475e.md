# EKS

Alternatives Names: Elastic Kubernetes Service; AWS EKS; Managed Kubernetes
Resource Link: https://docs.aws.amazon.com/eks
Category: Container Orchestration
Programming Language: YAML (config) / Python/Go (clients)
Description: AWS managed Kubernetes service. Handles control plane provisioning, upgrades, and availability. Runs containerized workloads with auto-scaling, rolling deployments, per-namespace isolation, and service mesh support.
Role in VC Assistant Infrastructure: Runs production platform services: FastAPI orchestration pods, document pipeline workers, reranking services, and vLLM inference servers. Per-tenant Kubernetes namespaces provide workload isolation. HPA auto-scales API pods during pitch deck processing peaks.
Potential Alternatives (with main comparison): ECS: simpler AWS-native orchestration with less ops overhead — recommended before EKS for small teams. GKE (Google Kubernetes Engine): better Kubernetes experience than EKS but locks you to GCP. AKS (Azure): similar to EKS but Azure ecosystem. Self-managed K8s: full control but high ops burden — never worth it for a startup. ECS Fargate: serverless containers, good migration path from ECS to EKS.
Deployment Notes & File Structure: Recommended structure:
k8s/
  namespaces/
    tenant-template.yaml  # namespace + RBAC per tenant
  deployments/
    vc-api.yaml
    doc-pipeline.yaml
    vllm-server.yaml
  services/
    vc-api-svc.yaml
  ingress/
    alb-ingress.yaml
  helm/
    vc-platform/          # umbrella Helm chart
Short File/Config Template: # k8s/deployments/vc-api.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: vc-api
  namespace: tenant-{{ TENANT_ID }}
spec:
  replicas: 2
  template:
    spec:
      containers:
      - name: api
        image: your-ecr/vc-api:latest
Stack & Framework Integration: Provisioned by Terraform (eks_cluster module). Applications deployed via Helm charts or kubectl manifests. Each tenant gets a dedicated namespace. Ingress via AWS ALB Ingress Controller. Secrets managed via AWS Secrets Manager + External Secrets Operator. CI/CD via GitHub Actions + Helm upgrade.

# Stack & Framework Integration

Provisioned by Terraform (eks_cluster module). Applications deployed via Helm charts or kubectl manifests. Each tenant gets a dedicated namespace. Ingress via AWS ALB Ingress Controller. Secrets managed via AWS Secrets Manager + External Secrets Operator. CI/CD via GitHub Actions + Helm upgrade.

# Alternative Tools

ECS: simpler AWS-native orchestration with less ops overhead — recommended before EKS for small teams. GKE (Google Kubernetes Engine): better Kubernetes experience than EKS but locks you to GCP. AKS (Azure): similar to EKS but Azure ecosystem. Self-managed K8s: full control but high ops burden — never worth it for a startup. ECS Fargate: serverless containers, good migration path from ECS to EKS.
